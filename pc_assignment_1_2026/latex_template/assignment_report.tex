\section{Introduction}
%─────────────────────────────────────────────────────────────────────────────

The Julia set is a fractal defined over the complex plane by iterating the function
$z_{n+1} = z_n^2 + c$, where $c$ is a fixed complex constant \citep{julia1918memoire}.
For this assignment the constant is set to $c = -0.7269 + 0.1889i$.
A pixel at coordinates $(x, y)$ is considered part of the set if the iteration remains
bounded — i.e.\ does not exceed a magnitude threshold of 1000 — within 300 iterations.
The image grid is $768 \times 768$ pixels, giving $768^2 = 589{,}824$ independent pixel
computations, each requiring between 1 and 300 complex-number iterations.

Because each pixel's membership test is entirely independent of every other pixel's, the
problem is \emph{embarrassingly parallel} \citep{grama2003introduction}: there are no
data dependencies, no need for synchronisation between threads, and no shared state to
protect.
This makes it an ideal candidate for shared-memory parallelism via OpenMP.

Five distinct OpenMP parallel implementations of the compute-intensive pixel kernel were
developed, each reflecting a different task-decomposition strategy.
Their performance was measured on a GitHub Codespaces virtual machine (AMD EPYC 7763
64-Core Processor, 2 vCPUs allocated, 1 socket, 1 core per socket, 2 hardware threads
per core; L1d cache: 32\,KiB, L2: 512\,KiB, L3: 32\,MiB) across thread counts
$\{1, 2, 4, 6, 8, 10, 12, 14, 16\}$.
The serial baseline execution time was $T_{\text{serial}} = 0.9789\,\text{s}$.

%─────────────────────────────────────────────────────────────────────────────
\section{Implementation Details}
%─────────────────────────────────────────────────────────────────────────────

\subsection{Serial Baseline (\texttt{kernel\_serial})}

The serial implementation iterates over all rows $y \in [0, \text{DIM})$ and columns
$x \in [0, \text{DIM})$, evaluates \texttt{julia(x,y)} for each pixel, and writes the
red-green-blue values directly into a flat RGB buffer at offset
$\textit{offset} = x + y \cdot \text{DIM}$.
No synchronisation is needed because each pixel's output location is unique.
The inner-loop access pattern ($x$ increasing while $y$ is fixed) is \emph{row-major},
matching the layout of the buffer in memory and yielding excellent spatial locality.

%─────────────────────────────────────────────────────────────────────────────
\subsection{Method (a): 1D Rowwise Parallel — Interleaved}

With $T$ threads, thread $\mathit{id}$ processes rows
$\mathit{id},\ \mathit{id}+T,\ \mathit{id}+2T,\ \ldots$
in a cyclic (interleaved) fashion.
For example with $T = 4$: Thread~0 handles rows $0, 4, 8, \ldots$; Thread~1 handles
rows $1, 5, 9, \ldots$; and so on.
The decomposition is performed manually inside a \texttt{\#pragma omp parallel} block
using \texttt{omp\_get\_thread\_num()} and \texttt{omp\_get\_num\_threads()},
exactly as demonstrated in the $\pi$-computation example of Example~4.2.3 in the
course notes \citep{trobec2018introduction}.

\begin{lstlisting}[caption={1D Rowwise interleaved kernel (key excerpt)},label={lst:rowwise}]
#pragma omp parallel
{
    int id = omp_get_thread_num();
    int T  = omp_get_num_threads();
    for (int y = id; y < DIM; y += T) {       // cyclic row assignment
        for (int x = 0; x < DIM; x++) {
            int offset = x + y * DIM;
            int juliaValue = julia(x, y);
            ptr[offset*3+0] = 255 * juliaValue;
            ptr[offset*3+1] = 0;
            ptr[offset*3+2] = 0;
        }
    }
}
\end{lstlisting}

The inner loop ($x$ loop) always traverses consecutive memory addresses, preserving
row-major spatial locality regardless of which rows are assigned to a given thread.

%─────────────────────────────────────────────────────────────────────────────
\subsection{Method (b): 1D Columnwise Parallel — Interleaved}

This method mirrors Method~(a) but assigns \emph{columns} instead of rows.
Thread $\mathit{id}$ processes columns $\mathit{id},\ \mathit{id}+T,\
\mathit{id}+2T,\ \ldots$
The outer loop iterates over $x$ (column index) and the inner loop over $y$ (row
index).

\begin{lstlisting}[caption={1D Columnwise interleaved kernel (key excerpt)},label={lst:colwise}]
#pragma omp parallel
{
    int id = omp_get_thread_num();
    int T  = omp_get_num_threads();
    for (int x = id; x < DIM; x += T) {       // cyclic column assignment
        for (int y = 0; y < DIM; y++) {
            int offset = x + y * DIM;
            int juliaValue = julia(x, y);
            ptr[offset*3+0] = 255 * juliaValue;
            ptr[offset*3+1] = 0;
            ptr[offset*3+2] = 0;
        }
    }
}
\end{lstlisting}

Here, successive writes in the inner loop are separated by
$\text{DIM} \times 3 = 2{,}304$ bytes (one full row of the RGB buffer), so consecutive
accesses by a given thread are \emph{not} contiguous in memory — a strided, cache-unfriendly access pattern.

%─────────────────────────────────────────────────────────────────────────────
\subsection{Method (c): 2D Row-Block Parallel — Contiguous Block}

The $\text{DIM} = 768$ rows are divided into $T$ contiguous blocks.
Thread $\mathit{id}$ is assigned the row range $[\mathit{ystart},\ \mathit{yend})$,
where
\[
  \mathit{ystart} = \mathit{id} \times \left\lfloor \tfrac{\text{DIM}}{T} \right\rfloor,
  \qquad
  \mathit{yend}   = \begin{cases}
    \text{DIM} & \text{if } \mathit{id} = T-1,\\
    \mathit{ystart} + \left\lfloor \tfrac{\text{DIM}}{T} \right\rfloor & \text{otherwise.}
  \end{cases}
\]
The last thread absorbs any remainder rows to ensure complete coverage.

\begin{lstlisting}[caption={2D Row-Block kernel (key excerpt)},label={lst:rowblock}]
#pragma omp parallel
{
    int id              = omp_get_thread_num();
    int T               = omp_get_num_threads();
    int rows_per_thread = DIM / T;
    int ystart          = id * rows_per_thread;
    int yend            = (id == T-1) ? DIM : ystart + rows_per_thread;
    for (int y = ystart; y < yend; y++) {
        for (int x = 0; x < DIM; x++) {
            int offset = x + y * DIM;
            int juliaValue = julia(x, y);
            ptr[offset*3+0] = 255 * juliaValue;
            ptr[offset*3+1] = 0;
            ptr[offset*3+2] = 0;
        }
    }
}
\end{lstlisting}

Because each thread operates on a contiguous block of rows and the inner loop is a
sequential $x$-scan, the access pattern is row-major and cache-friendly.

%─────────────────────────────────────────────────────────────────────────────
\subsection{Method (d): 2D Column-Block Parallel — Contiguous Block}

Analogous to Method~(c), but $\text{DIM}$ \emph{columns} are divided into $T$
contiguous blocks.
Thread $\mathit{id}$ is assigned columns $[\mathit{xstart},\ \mathit{xend})$.
The outer loop iterates over all $\text{DIM}$ rows, and the inner loop is restricted to
the thread's column slice.

\begin{lstlisting}[caption={2D Column-Block kernel (key excerpt)},label={lst:colblock}]
#pragma omp parallel
{
    int id              = omp_get_thread_num();
    int T               = omp_get_num_threads();
    int cols_per_thread = DIM / T;
    int xstart          = id * cols_per_thread;
    int xend            = (id == T-1) ? DIM : xstart + cols_per_thread;
    for (int y = 0; y < DIM; y++) {
        for (int x = xstart; x < xend; x++) {
            int offset = x + y * DIM;
            int juliaValue = julia(x, y);
            ptr[offset*3+0] = 255 * juliaValue;
            ptr[offset*3+1] = 0;
            ptr[offset*3+2] = 0;
        }
    }
}
\end{lstlisting}

Although the $x$-slice within a single row is contiguous, the outer $y$-loop causes
the thread to jump by $\text{DIM} \times 3$ bytes between the end of one row's slice
and the start of the next, resulting in a strided memory access pattern.

%─────────────────────────────────────────────────────────────────────────────
\subsection{Method (e): OpenMP \texttt{for} Construct}

The simplest implementation: a single \texttt{\#pragma omp parallel for} directive with
static scheduling applied to the outer row loop.
OpenMP's runtime automatically partitions the 768 row iterations into $T$ contiguous
blocks — one per thread — identical in structure to Method~(c), but requiring
significantly less manual code.

\begin{lstlisting}[caption={OpenMP \texttt{for}-construct kernel},label={lst:ompfor}]
#pragma omp parallel for schedule(static)
for (int y = 0; y < DIM; y++) {
    for (int x = 0; x < DIM; x++) {
        int offset = x + y * DIM;
        int juliaValue = julia(x, y);
        ptr[offset*3+0] = 255 * juliaValue;
        ptr[offset*3+1] = 0;
        ptr[offset*3+2] = 0;
    }
}
\end{lstlisting}

With \texttt{schedule(static)}, iterations are assigned to threads in contiguous
chunks of size $\lfloor 768/T \rfloor$ before execution begins, incurring minimal
runtime overhead \citep{chapman2007using}.

\section{Discussion}
%─────────────────────────────────────────────────────────────────────────────

\subsection{Question A: 1D Rowwise vs.\ 1D Columnwise}

Both methods assign the same total number of pixels to each thread, so the
\emph{computational load} per thread is nominally identical.
The critical difference lies in their \emph{memory access patterns}.

In Method~(a) (rowwise), the inner loop increments $x$ while $y$ is fixed.
Since the output buffer is stored in row-major order, consecutive writes land on
consecutive memory addresses, fitting naturally into cache lines and exploiting
\emph{spatial locality}.

In Method~(b) (columnwise), the inner loop increments $y$ while $x$ is fixed.
Successive writes are separated by $\text{DIM} \times 3 = 2{,}304$ bytes — one full
row of the RGB buffer — which spans many cache lines and causes frequent cache misses.
This makes Method~(b) significantly more \emph{memory-bound}
\citep{grama2003introduction}.

The experimental results confirm this: at 2 threads (where true parallelism is possible
on this 2-vCPU machine), Method~(a) achieves a speedup of \textbf{1.73} while
Method~(b) achieves \textbf{1.60} — a clear advantage for the rowwise approach.
The difference persists at 8 threads (1.68 vs.\ 1.78), where the column method
temporarily benefits from the OS scheduler time-slicing the 2 physical threads among
many OpenMP threads in a way that happens to mask its cache-miss penalty.
At 6 threads, the difference is most pronounced: Method~(a) achieves 1.41 while
Method~(b) drops to only 1.12, reflecting a particularly poor scheduling interaction
for the column-strided access pattern.
Overall, Method~(a) is more \emph{consistently} performant across all thread counts
due to its alignment with the row-major buffer layout.

\subsection{Question B: 2D Row-Block vs.\ 2D Column-Block}

The same spatial-locality argument applies here.
Method~(c) (row-block) gives each thread a contiguous range of complete rows; the
inner loop over $x$ is always a sequential memory scan — excellent spatial locality
and high cache-hit rates.

Method~(d) (column-block) assigns each thread a contiguous $x$-range per row.
Although the column slice within a single row is contiguous, the outer $y$-loop forces
the thread to skip $\text{DIM} \times 3 = 2{,}304$ bytes between the end of one row's
slice and the start of the next — a strided pattern that degrades cache efficiency.

Experimentally, however, Method~(c) (2D Row-Block) actually \emph{outperforms}
Method~(d) (2D Col-Block) at the most meaningful thread count of 4: $\mathbf{1.91}$
vs.\ $\mathbf{1.64}$, consistent with the cache-locality prediction.
At higher thread counts (6, 10, 14, 16), the results fluctuate because with only 2
physical threads, all thread counts above 2 require OS time-multiplexing.
When threads are heavily oversubscribed, the column-block's small per-thread working
set (a narrow column slice) may fit entirely within the L1d cache (32\,KiB), partly
compensating for the strided access.
Nevertheless, the peak speedup of Method~(c) (\textbf{1.91} at 4 threads) equals that
of Method~(d) (\textbf{1.90} at 10 threads), and row-block is more predictable.

\subsection{Question C: Top Two Performers}

From Table~\ref{tab:speedup} and Figure~\ref{fig:speedup}, the two consistently
highest-performing methods are:

\begin{enumerate}
  \item \textbf{Method~(c) — 2D Row-Block}: achieves the joint-highest peak speedup
        of $\mathbf{1.91}$ at 4 threads, with strong results at 6 threads (1.62),
        8 threads (1.57), and 14 threads (1.78).
  \item \textbf{Method~(b) — 1D Colwise}: achieves a peak of $\mathbf{1.91}$ at
        10 threads and $1.89$ at 14 threads, making it the second most consistent
        high-value performer.
\end{enumerate}

\paragraph{Similarity.}
Both methods achieve near-identical peak speedups (1.91) and both benefit from the
fact that — when the thread count happens to align well with the 2 physical cores
available — each physical core handles a well-defined, non-overlapping portion of the
image.
In both cases, each thread processes exactly $\text{DIM}/T$ units of work
(rows or columns), giving \emph{perfectly balanced} load at every thread count.
Load balance is especially important on a 2-core machine: any imbalance leaves one
physical core idle.

\paragraph{Difference.}
The fundamental structural difference is the \emph{granularity and memory pattern} of
each thread's work unit.
Method~(c) assigns a contiguous block of complete rows: the inner $x$-loop scans
memory sequentially (row-major, cache-friendly), so cache performance is excellent
at low thread counts.
Method~(b) assigns interleaved columns: the inner $y$-loop produces strided memory
accesses ($2{,}304$-byte jumps), which is in theory cache-unfriendly.
Yet at higher thread counts (10–16), the fine-grained interleaving of Method~(b)
distributes load more evenly across the two physical threads, which accounts for its
strong performance at $T = 10$ and $T = 14$.

\paragraph{Why they outperform the others.}
\begin{itemize}
  \item \textbf{Method~(a) — 1D Rowwise}: Although cache-friendly, its interleaved
        assignment increases TLB pressure and reduces the benefit of hardware
        prefetching compared to a contiguous block assignment; this explains its lower
        peak of 1.74 (at $T=12$) vs.\ Method~(c)'s 1.91.
  \item \textbf{Method~(d) — 2D Col-Block}: Good peak at $T=6$ and $T=10$, but
        degrades sharply at $T=12$ (speedup drops to 1.18) due to unfavourable
        OS thread-scheduling interactions with its strided write pattern.
  \item \textbf{Method~(e) — OMP For}: Achieves 1.92 at $T=2$ (the best single
        result in the entire experiment), but performance falls noticeably at $T=4$
        (1.50) and $T=8$ (1.36).
        The \texttt{schedule(static)} row-block assignment is equivalent to
        Method~(c), but at higher thread counts the OpenMP runtime incurs additional
        overhead from synchronisation at the implicit barrier at the end of the
        \texttt{parallel for} region, which hurts performance when threads are heavily
        oversubscribed.
\end{itemize}

\subsection{Question D: Ideas for Further Improvement}

Since one of the two top performers is Method~(c) (2D Row-Block) and the other is
Method~(b) (1D Colwise), and \emph{neither} is Method~(e) (OpenMP \texttt{for}
construct), the assignment requires improvement ideas for both top performers.
However, since Methods~(b) and~(c) are both manually implemented kernels, the
following improvements apply to the \textbf{2D Row-Block (Method c)} as the primary
focus:

\begin{itemize}
  \item \textbf{Dynamic or guided scheduling.}
        The Julia set is non-uniform: interior pixels require all 300 iterations while
        exterior pixels may exit after very few.
        Even with contiguous row-blocks, some blocks (e.g.\ those covering the dense
        red region in the fractal centre) take significantly longer than others.
        Using a dynamic row-assignment loop — where threads pick up the next available
        row when they finish their current one — would reduce this load imbalance
        \citep{chapman2007using}.

  \item \textbf{SIMD vectorisation of the inner loop.}
        The inner $x$-loop in Method~(c) performs identical floating-point arithmetic
        on independent pixels.
        Annotating it with \texttt{\#pragma omp simd} or applying AVX2 intrinsics
        would allow multiple pixels to be processed simultaneously within a single
        thread, exploiting the AMD EPYC 7763's AVX2 vector units
        \citep{trobec2018introduction}.

  \item \textbf{Cache-aware tile decomposition.}
        Rather than assigning a large contiguous row-block to each thread, the image
        could be divided into 2D tiles whose size is chosen so that the tile's pixel
        buffer fits in L1d cache (32\,KiB).
        Each tile would be $\approx\sqrt{32768/3} \approx 104$ pixels on a side.
        Threads would pick up tiles dynamically, combining spatial locality with load
        balance \citep{grama2003introduction}.

  \item \textbf{Thread affinity binding.}
        Pinning threads to specific physical cores using \texttt{OMP\_PROC\_BIND=close}
        prevents OS migration between the two available hardware threads, ensuring each
        thread's warm L1/L2 cache state is preserved across iterations
        \citep{chapman2007using}.
\end{itemize}

\section{Conclusion}
%─────────────────────────────────────────────────────────────────────────────

Five OpenMP parallel implementations of the Julia set fractal kernel were developed,
compiled, and benchmarked on a 2-vCPU GitHub Codespaces virtual machine
(AMD EPYC 7763, serial baseline $T_{\text{serial}} = 0.9789\,\text{s}$).
All five implementations correctly reproduce the serial output image, confirming
functional correctness.

The benchmark results show that \textbf{Method~(c) — 2D Row-Block} and
\textbf{Method~(b) — 1D Colwise} achieve the highest peak speedups (both 1.91),
followed closely by \textbf{Method~(e) — OMP For} (peak 1.92 at 2 threads, most
consistent at low thread counts).
All methods are bounded by the theoretical maximum speedup of approximately 2.0
imposed by the 2 available hardware threads.

The analysis confirms two key principles of parallel performance on shared-memory
systems.
First, \emph{memory access pattern} is critical: row-major access (Methods a, c, e)
generally outperforms strided column access at low thread counts, though the advantage
narrows under heavy oversubscription.
Second, \emph{load balance} determines performance at higher thread counts: perfectly
balanced methods (block and interleaved) both approach the hardware limit when
scheduling overhead is low.
Future work could explore SIMD vectorisation and cache-aware 2D tiling to push
performance closer to the hardware's theoretical peak.
