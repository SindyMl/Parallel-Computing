\section{Introduction}
%─────────────────────────────────────────────────────────────────────────────

The Julia set is a fractal defined over the complex plane by iterating the function
$z_{n+1} = z_n^2 + c$, where $c$ is a fixed complex constant \citep{julia1918memoire}.
For this assignment the constant is set to $c = -0.7269 + 0.1889i$.
A pixel at coordinates $(x, y)$ is considered part of the set if the iteration remains
bounded — i.e.\ does not exceed a magnitude threshold of 1000 — within 300 iterations.
The image grid is $768 \times 768$ pixels, giving $768^2 = 589{,}824$ independent pixel
computations, each requiring between 1 and 300 complex-number iterations.

Because each pixel's membership test is entirely independent of every other pixel's, the
problem is \emph{embarrassingly parallel} \citep{grama2003introduction}: there are no
data dependencies, no need for synchronisation between threads, and no shared state to
protect.
This makes it an ideal candidate for shared-memory parallelism via OpenMP.

Five distinct OpenMP parallel implementations of the compute-intensive pixel kernel were
developed, each reflecting a different task-decomposition strategy.
Their performance was measured on a GitHub Codespaces virtual machine (AMD EPYC 7763
64-Core Processor, 2 vCPUs allocated, 1 socket, 1 core per socket, 2 hardware threads
per core; L1d cache: 32\,KiB, L2: 512\,KiB, L3: 32\,MiB) across thread counts
$\{1, 2, 4, 6, 8, 10, 12, 14, 16\}$.
The serial baseline execution time was $T_{\text{serial}} = 0.9789\,\text{s}$.

%─────────────────────────────────────────────────────────────────────────────
\section{Implementation Details}
%─────────────────────────────────────────────────────────────────────────────

\subsection{Serial Baseline (\texttt{kernel\_serial})}

The serial implementation iterates over all rows $y \in [0, \text{DIM})$ and columns
$x \in [0, \text{DIM})$, evaluates \texttt{julia(x,y)} for each pixel, and writes the
red-green-blue values directly into a flat RGB buffer at offset
$\textit{offset} = x + y \cdot \text{DIM}$.
No synchronisation is needed because each pixel's output location is unique.
The inner-loop access pattern ($x$ increasing while $y$ is fixed) is \emph{row-major},
matching the layout of the buffer in memory and yielding excellent spatial locality.

%─────────────────────────────────────────────────────────────────────────────
\subsection{Method (a): 1D Rowwise Parallel — Interleaved}

With $T$ threads, thread $\mathit{id}$ processes rows
$\mathit{id},\ \mathit{id}+T,\ \mathit{id}+2T,\ \ldots$
in a cyclic (interleaved) fashion.
For example with $T = 4$: Thread~0 handles rows $0, 4, 8, \ldots$; Thread~1 handles
rows $1, 5, 9, \ldots$; and so on.
The decomposition is performed manually inside a \texttt{\#pragma omp parallel} block
using \texttt{omp\_get\_thread\_num()} and \texttt{omp\_get\_num\_threads()},
exactly as demonstrated in the $\pi$-computation example of Example~4.2.3 in the
course notes \citep{trobec2018introduction}.

\begin{lstlisting}[caption={1D Rowwise interleaved kernel (key excerpt)},label={lst:rowwise}]
#pragma omp parallel
{
    int id = omp_get_thread_num();
    int T  = omp_get_num_threads();
    for (int y = id; y < DIM; y += T) {       // cyclic row assignment
        for (int x = 0; x < DIM; x++) {
            int offset = x + y * DIM;
            int juliaValue = julia(x, y);
            ptr[offset*3+0] = 255 * juliaValue;
            ptr[offset*3+1] = 0;
            ptr[offset*3+2] = 0;
        }
    }
}
\end{lstlisting}

The inner loop ($x$ loop) always traverses consecutive memory addresses, preserving
row-major spatial locality regardless of which rows are assigned to a given thread.

%─────────────────────────────────────────────────────────────────────────────
\subsection{Method (b): 1D Columnwise Parallel — Interleaved}

This method mirrors Method~(a) but assigns \emph{columns} instead of rows.
Thread $\mathit{id}$ processes columns $\mathit{id},\ \mathit{id}+T,\
\mathit{id}+2T,\ \ldots$
The outer loop iterates over $x$ (column index) and the inner loop over $y$ (row
index).

\begin{lstlisting}[caption={1D Columnwise interleaved kernel (key excerpt)},label={lst:colwise}]
#pragma omp parallel
{
    int id = omp_get_thread_num();
    int T  = omp_get_num_threads();
    for (int x = id; x < DIM; x += T) {       // cyclic column assignment
        for (int y = 0; y < DIM; y++) {
            int offset = x + y * DIM;
            int juliaValue = julia(x, y);
            ptr[offset*3+0] = 255 * juliaValue;
            ptr[offset*3+1] = 0;
            ptr[offset*3+2] = 0;
        }
    }
}
\end{lstlisting}

Here, successive writes in the inner loop are separated by
$\text{DIM} \times 3 = 2{,}304$ bytes (one full row of the RGB buffer), so consecutive
accesses by a given thread are \emph{not} contiguous in memory — a strided, cache-
unfriendly access pattern.

%─────────────────────────────────────────────────────────────────────────────
\subsection{Method (c): 2D Row-Block Parallel — Contiguous Block}

The $\text{DIM} = 768$ rows are divided into $T$ contiguous blocks.
Thread $\mathit{id}$ is assigned the row range $[\mathit{ystart},\ \mathit{yend})$,
where
\[
  \mathit{ystart} = \mathit{id} \times \left\lfloor \tfrac{\text{DIM}}{T} \right\rfloor,
  \qquad
  \mathit{yend}   = \begin{cases}
    \text{DIM} & \text{if } \mathit{id} = T-1,\\
    \mathit{ystart} + \left\lfloor \tfrac{\text{DIM}}{T} \right\rfloor & \text{otherwise.}
  \end{cases}
\]
The last thread absorbs any remainder rows to ensure complete coverage.

\begin{lstlisting}[caption={2D Row-Block kernel (key excerpt)},label={lst:rowblock}]
#pragma omp parallel
{
    int id              = omp_get_thread_num();
    int T               = omp_get_num_threads();
    int rows_per_thread = DIM / T;
    int ystart          = id * rows_per_thread;
    int yend            = (id == T-1) ? DIM : ystart + rows_per_thread;
    for (int y = ystart; y < yend; y++) {
        for (int x = 0; x < DIM; x++) {
            int offset = x + y * DIM;
            int juliaValue = julia(x, y);
            ptr[offset*3+0] = 255 * juliaValue;
            ptr[offset*3+1] = 0;
            ptr[offset*3+2] = 0;
        }
    }
}
\end{lstlisting}

Because each thread operates on a contiguous block of rows and the inner loop is a
sequential $x$-scan, the access pattern is row-major and cache-friendly.

%─────────────────────────────────────────────────────────────────────────────
\subsection{Method (d): 2D Column-Block Parallel — Contiguous Block}

Analogous to Method~(c), but $\text{DIM}$ \emph{columns} are divided into $T$
contiguous blocks.
Thread $\mathit{id}$ is assigned columns $[\mathit{xstart},\ \mathit{xend})$.
The outer loop iterates over all $\text{DIM}$ rows, and the inner loop is restricted to
the thread's column slice.

\begin{lstlisting}[caption={2D Column-Block kernel (key excerpt)},label={lst:colblock}]
#pragma omp parallel
{
    int id              = omp_get_thread_num();
    int T               = omp_get_num_threads();
    int cols_per_thread = DIM / T;
    int xstart          = id * cols_per_thread;
    int xend            = (id == T-1) ? DIM : xstart + cols_per_thread;
    for (int y = 0; y < DIM; y++) {
        for (int x = xstart; x < xend; x++) {
            int offset = x + y * DIM;
            int juliaValue = julia(x, y);
            ptr[offset*3+0] = 255 * juliaValue;
            ptr[offset*3+1] = 0;
            ptr[offset*3+2] = 0;
        }
    }
}
\end{lstlisting}

Although the $x$-slice within a single row is contiguous, the outer $y$-loop causes
the thread to jump by $\text{DIM} \times 3$ bytes between the end of one row's slice
and the start of the next, resulting in a strided memory access pattern.

%─────────────────────────────────────────────────────────────────────────────
\subsection{Method (e): OpenMP \texttt{for} Construct}

The simplest implementation: a single \texttt{\#pragma omp parallel for} directive with
static scheduling applied to the outer row loop.
OpenMP's runtime automatically partitions the 768 row iterations into $T$ contiguous
blocks — one per thread — identical in structure to Method~(c), but requiring
significantly less manual code.

\begin{lstlisting}[caption={OpenMP \texttt{for}-construct kernel},label={lst:ompfor}]
#pragma omp parallel for schedule(static)
for (int y = 0; y < DIM; y++) {
    for (int x = 0; x < DIM; x++) {
        int offset = x + y * DIM;
        int juliaValue = julia(x, y);
        ptr[offset*3+0] = 255 * juliaValue;
        ptr[offset*3+1] = 0;
        ptr[offset*3+2] = 0;
    }
}
\end{lstlisting}

With \texttt{schedule(static)}, iterations are assigned to threads in contiguous
chunks of size $\lfloor 768/T \rfloor$ before execution begins, incurring minimal
runtime overhead \citep{chapman2007using}.

\section{Discussion}
%─────────────────────────────────────────────────────────────────────────────

\subsection{Question A: 1D Rowwise vs.\ 1D Columnwise}

Both methods assign the same total number of pixels to each thread, so the
\emph{computational load} per thread is nominally equal.
The difference lies in their \emph{memory access patterns}.

In Method~(a) (rowwise), the inner loop increments $x$ while $y$ is fixed.
Since the output buffer is stored in row-major order, consecutive writes land on
consecutive memory addresses, fitting naturally into cache lines and exploiting
\emph{spatial locality}.

In Method~(b) (columnwise), the inner loop increments $y$ while $x$ is fixed.
Successive writes are separated by $\text{DIM} \times 3 = 2{,}304$ bytes — one full
row of the RGB buffer — which spans many cache lines and causes frequent cache misses.
This makes Method~(b) significantly more \emph{memory-bound}
\citep{grama2003introduction}.

The experimental results reflect this: at 2 threads (where parallelism is most
meaningful on this 2-vCPU machine), Method~(a) achieves a speedup of 1.73 while
Method~(b) achieves 1.60.
At higher thread counts (e.g.\ 10 threads), the gap reverses somewhat because the OS
scheduler is forced to time-share the 2 physical threads among many OpenMP threads,
introducing context-switch overhead that masks the cache-miss penalty of the column
method.
Overall, the 1D rowwise method is more consistently performant due to better cache
utilisation.

\subsection{Question B: 2D Row-Block vs.\ 2D Column-Block}

The same cache-locality argument applies.
Method~(c) (row-block) gives each thread a contiguous range of complete rows; the
inner loop over $x$ is always a sequential memory scan — excellent spatial locality.

Method~(d) (column-block) gives each thread a contiguous $x$-range within every row.
Although the slice within a single row is contiguous, the outer $y$-loop forces the
thread to skip $\text{DIM} \times 3 = 2{,}304$ bytes between the end of one row's
slice and the start of the next.
This strided pattern degrades cache performance.

Experimentally, however, Method~(d) (2D Col-Block) frequently \emph{outperforms}
Method~(c) (2D Row-Block) on this 2-vCPU machine — for example, at 4 threads:
$1.91$ vs.\ $1.64$.
This seemingly counter-intuitive result arises because with only 2 hardware threads,
all thread counts above 2 require \emph{time-multiplexing}.
When many threads share 2 physical cores, the kernel's short inner loop per column
slice tends to interleave well with the CPU's out-of-order execution pipeline, masking
latency.
Additionally, the column-block decomposition causes each thread to touch a small,
fixed set of output columns, which may fit well in the L1/L2 cache on this particular
processor (L1d: 32\,KiB per core), partially mitigating the strided access penalty.

\subsection{Question C: Top Two Performers}

From Table~\ref{tab:speedup} and Figure~\ref{fig:speedup}, the two consistently
highest-performing methods are:

\begin{enumerate}
  \item \textbf{Method~(e) — OpenMP \texttt{for} construct}: achieves the highest
        peak speedup of $\mathbf{1.95}$ at 4 threads and $1.94$ at 2 threads.
  \item \textbf{Method~(d) — 2D Column-Block}: achieves a peak of $1.91$ at 4 threads
        and $1.90$ at 10 threads, the second most consistent top performer.
\end{enumerate}

\paragraph{Similarity.}
Both methods involve contiguous allocation of work to each thread.
Method~(e) uses OpenMP's \texttt{schedule(static)} to assign contiguous row-blocks,
and Method~(d) assigns contiguous column-blocks.
In both cases, the amount of work per thread is \emph{perfectly balanced} — each
thread processes exactly $\text{DIM}/T$ units (rows or columns).
Perfect load balance is critical on this machine: with only 2 physical threads, any
imbalance would cause one physical thread to sit idle while the other is still
computing.

\paragraph{Difference.}
The key structural difference is the loop order.
Method~(e) iterates rows in the outer loop and columns in the inner loop (row-major,
cache-friendly).
Method~(d) iterates all rows in the outer loop but restricts the inner loop to a
column slice (strided access, cache-unfriendly in theory).
Despite this, Method~(d) performs well because on this specific machine the column
slices are small enough to reside in L1 cache for the duration of each thread's inner
computation.

\paragraph{Why they outperform the others.}
The remaining three methods suffer from at least one of:
\begin{itemize}
  \item \textbf{Interleaved row/column assignment} (Methods a, b): interleaved access
        increases TLB pressure and reduces the benefit of hardware prefetching compared
        to contiguous block assignment.
  \item \textbf{Poor cache locality} (Method b): strided inner-loop writes cause
        cache-line thrashing.
  \item \textbf{Uneven effective load at high thread counts} (Methods c and a at 16
        threads): when $T \gg$ number of physical cores, the overhead of managing many
        idle threads outweighs the benefit.
\end{itemize}
Method~(e) benefits additionally from OpenMP's internal optimisations: a persistent
thread pool (avoiding thread creation overhead on each call), and compiler-assisted
loop optimisations enabled by the \texttt{parallel for} directive
\citep{chapman2007using}.

\subsection{Question D: Ideas for Further Improvement}

Since one of the two top performers is Method~(e) (OpenMP \texttt{for} construct),
the assignment requires improvement ideas only for the \emph{other} top performer —
\textbf{Method~(d): 2D Column-Block}.
The following improvements are proposed:

\begin{itemize}
  \item \textbf{Dynamic or guided scheduling within the column block.}
        The Julia set is non-uniform: pixels in the set's interior require all 300
        iterations, while exterior pixels may exit after very few.
        Rather than statically assigning a fixed column block to each thread, using
        \texttt{schedule(dynamic)} with a small chunk size would allow faster threads
        to pick up additional column slices, reducing load imbalance
        \citep{chapman2007using}.

  \item \textbf{SIMD vectorisation of the inner loop.}
        The $y$-loop inside each thread's column slice performs identical floating-point
        operations on independent data.
        Annotating it with \texttt{\#pragma omp simd} (or using AVX2 intrinsics
        directly) would allow multiple pixels to be computed per CPU clock cycle,
        exploiting the AMD EPYC's AVX2 vector units
        \citep{trobec2018introduction}.

  \item \textbf{Cache-aware column-slice sizing.}
        The column-block size $\lfloor \text{DIM}/T \rfloor$ is determined purely by
        the thread count.
        Choosing the slice width so that one thread's working set
        ($\text{width} \times \text{DIM} \times 3$ bytes) fits within the L1d cache
        (32\,KiB) would maximise cache reuse.
        For the 768-row image, this means a column-slice width of approximately
        $32768 / (768 \times 3) \approx 14$ columns.
        The number of threads could be tuned accordingly.

  \item \textbf{Thread affinity binding.}
        Pinning each thread to a specific hardware thread using
        \texttt{OMP\_PROC\_BIND=close} prevents the OS from migrating threads between
        physical cores, which would invalidate warm cache state and increase memory
        access latency \citep{grama2003introduction}.
\end{itemize}

\section{Conclusion}
%─────────────────────────────────────────────────────────────────────────────

Five OpenMP parallel implementations of the Julia set fractal kernel were developed,
compiled, and benchmarked on a 2-vCPU GitHub Codespaces virtual machine.
All five correctly reproduce the serial output image, verifying functional correctness.

The results demonstrate that the \textbf{OpenMP \texttt{for} construct} (Method~e)
and the \textbf{2D Column-Block} decomposition (Method~d) are the two consistently
best-performing approaches, both achieving peak speedups close to the theoretical
maximum of~2.0 imposed by the 2 available hardware threads.
Their success is attributable to balanced workload distribution and — in the case of
Method~(e) — additional runtime optimisations provided by the OpenMP library.

The analysis confirms that on shared-memory systems, \emph{memory access pattern}
is often as important as \emph{computation distribution}: row-based decompositions
with sequential inner loops exploit cache-line spatial locality, while column-based
methods with strided access can still perform well when the working set fits within
the L1 cache.
Future work could explore SIMD vectorisation and cache-aware tile sizes to push
performance closer to the hardware's theoretical peak.
